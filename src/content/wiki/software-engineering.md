Software engineering is not reducible to writing correct code. The craft encompasses reading tradeoffs, managing complexity over time, communicating across organizational boundaries, and maintaining systems under conditions that no algorithm interview ever simulated. [Fagner Brack's critique of algorithm interviews](/reading/2026-04/2026-04-30t155134-learn-algorithms-for-interviews-forget-them-for-work) makes this plain: the skills that get candidates hired — memorized graph traversals, rehearsed dynamic programming patterns — correlate poorly with production performance, which demands incremental shipping, messy real-world inputs, and judgment about what not to build.

The gap between code-writing and system-owning is a recurring thread. [Anton Zaides's unwritten laws](/reading/2026-06/2026-06-10t073045-the-unwritten-laws-of-software-engineering) distill hard-won production lessons — roll back before debugging, treat every external dependency as a future outage — that no specification captures but every senior engineer carries. That tacit layer is the subject of [cekrem's essay on Polanyi](/reading/2026-05/2026-05-19t110710-the-tacit-dimension-why-your-best-engineers-cant-tell-you): the most valuable engineering expertise is pattern recognition and design intuition that cannot be articulated, let alone transferred to AI tools, and can only be transmitted through apprenticeship. [Tuhin Nair](/reading/2026-05/2026-05-13t060018-why-senior-developers-fail-to-communicate-their-expertise) locates the same gap differently: senior engineers think in complexity management while the rest of the business thinks in uncertainty reduction, and bridging that translation gap is the real work of seniority.

Design principles fare no better when abstracted from their intent. [Henrique Teixeira's reading of SRP](/reading/2026-06/2026-06-04t073318-single-responsibility-the-distorted-principle) argues that "do only one thing" misreads the Single Responsibility Principle, which is actually about cohesive grouping under a single accountable responsibility. Over-granularizing classes violates the cognitive simplicity SRP is meant to create. The same logic runs through [Go Monk's case for deep modules](/reading/2026-05/2026-05-04t231343-ai-likes-deep-modules): small interfaces hiding large implementations reduce complexity for both humans and LLMs, while shallow modules multiply surface area without adding clarity.

Tooling shapes how engineers work as much as principles do. [Christian Hofstede-Kuhn on shell tricks](/reading/2026-04/2026-04-30t231815-shell-tricks-that-actually-make-life-easier-and-save-your) covers the compounding value of Readline bindings, brace expansion, and script safety flags — small fluency gains that accumulate across a career. [Ben Gesoff's Jujutsu workflow](/reading/2026-05/2026-05-31t164252-reviewing-large-changes-with-jujutsu) demonstrates how a different version control model — first-class conflict objects, automatic descendant rebasing — changes what kinds of review workflows are even possible. [Ally Piechowski's git log diagnostics](/reading/2026-06/2026-06-18t024208-the-git-commands-i-run-before-reading-any-code) show that churn hotspots, bus factor, and bug clusters are legible from commit history before reading a line of code.

Security is not separate from craft. [Dan Goodin's reporting on invisible Unicode supply-chain attacks](/reading/2026-04/2026-04-30t231634-supply-chain-attack-using-invisible-code-hits-github-and) documents 151 malicious npm packages that encoded payloads in variation-selector characters, undetectable by reviewers and static analysis while remaining executable at runtime. The attack surface is the review process itself.

Testing is where engineering intentions meet reality. [The Currents team on Playwright](/reading/2026-05/2026-05-05t135218-designing-playwright-tests-that-survive-ui-refactors) argues that test suites break during refactors not because of bad selector choices alone but because tests couple to implementation details rather than semantic roles. Their [staging vs. production framework](/reading/2026-05/2026-05-15t120337-playwright-testing-in-staging-vs-production) extends this into operational strategy. AI-generated tests compound the problem: [How To Test Frontend's catalog](/reading/2026-06/2026-06-22t185420-code-smells-when-you-get-ai-to-write-your-frontend-tests) of 20+ recurring AI test smells includes over-mocking, happy-path-only coverage, and tests written to match buggy implementations rather than intended behavior.

The AI dimension cuts across everything. [Lars Faye's argument against full agentic workflows](/reading/2026-04/2026-04-27t145041-agentic-coding-is-a-trap) is that they accelerate skill atrophy, invert priorities toward speed over understanding, and create vendor dependency. [Abednego Gomes](/reading/2026-05/2026-05-14t223612-the-perils-of-ai-to-the-software-engineering-profession) calls shipping AI-generated code without review categorically incompatible with safety-critical systems. [Yusuf Aytas](/reading/2026-05/2026-05-22t091746-when-code-is-cheap-does-quality-still-matter) reframes the problem: AI lowers the cost of producing code but not the cost of owning it, and LLMs can generate polished technical debt faster than any individual engineer. [Armin Ronacher](/reading/2026-06/2026-06-23t161552-the-coming-loop) warns that agent harness loops amplify LLMs' worst tendencies — defensive, opaque code — and risk producing codebases that require machine participation to maintain.

Not every source here argues against AI tooling. [Jane Street's Yaron Minsky](/reading/2026-06/2026-06-15t021106-formal-methods-and-the-future-of-programming) sees agentic coding as making formal verification newly cost-effective, both by lowering proof-writing costs and by creating demand for verification that tests alone cannot provide. [Werner Vogels](/reading/2026-06/2026-06-30t173037-a-return-to-two-pizza-culture) finds that compressed prototyping time justifies amending the Working Backwards process. [Sam Alba's CI triage work at PostHog](/reading/2026-04/2026-04-30t195531-what-ci-actually-looks-like-at-a-100-person-team) shows an AI agent handling 575K weekly jobs and 33M test executions at scale.

The organizational dimension is at least as important as the technical one. [The Typical Set](/reading/2026-05/2026-05-06t110728-the-bottleneck-was-never-the-code) argues the bottleneck was never code-writing but shared context, specification clarity, and management coherence — and that coding agents amplify whatever alignment an organization already has. [DHg on onboarding](/reading/2026-05/2026-05-08t112608-your-onboarding-is-a-hazing-ritual-and-you-call-it-agile) shows how dysfunctional practices disguised as agile process systematically set new engineers up to fail while making the dysfunction invisible to management. Platform reliability matters too: [David Bushell's case for migrating off GitHub](/reading/2026-05/2026-05-10t205349-github-is-sinking) is partly a tooling argument, but mostly a warning that engineers building on unreliable infrastructure inherit its failures.

Across all these sources, the consistent through-line is that engineering skill is not the same as coding output — and that most forces currently reshaping the field, from AI tools to broken hiring to platform decay, are optimizing for output while eroding the judgment that makes output worth anything.
