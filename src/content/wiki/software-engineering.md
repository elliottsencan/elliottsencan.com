Software engineering is not a single thing. It is the accumulated set of practices, judgments, and social arrangements that let people build software that works reliably, can be understood by others, and survives contact with changing requirements. The sources collected here touch nearly every layer of that: language and framework choices, design principles, testing strategy, version control, CI infrastructure, security, tooling, team dynamics, and the increasingly urgent question of what AI-assisted development does to the discipline.

On design principles, the through-line is that good engineering is about managing complexity. ["AI Likes Deep Modules"](/reading/2026-05/2026-05-04t231343-ai-likes-deep-modules) argues, drawing on John Ousterhout's work, that small interfaces hiding large implementations reduce cognitive load for both humans and LLMs. ["Single Responsibility, the Distorted Principle"](/reading/2026-06/2026-06-04t073318-single-responsibility-the-distorted-principle) corrects a common misreading: SRP is about cohesive grouping of behavior under one accountable concern, not about splitting every function into its smallest possible unit. Both articles converge on the same point: over-granularization creates its own complexity. ["A Better Way to Build Angular Components"](/reading/2026-04/2026-04-30t232001-a-better-way-to-build-angular-components-from-inputs-to) applies this to UI components specifically, arguing that directive-based composition is preferable to adding ever more inputs to a single bloated component.

Testing runs through many of the sources as a persistent concern. ["Designing Playwright Tests That Survive UI Refactors"](/reading/2026-05/2026-05-05t135218-designing-playwright-tests-that-survive-ui-refactors) argues that brittle tests result not primarily from selector choices but from coupling to implementation details rather than semantic roles. ["Playwright Testing in Staging vs Production"](/reading/2026-05/2026-05-15t120337-playwright-testing-in-staging-vs-production) adds a layer on where tests belong in the deployment pipeline. ["Testing a Security Tool Like It Can Hurt People"](/reading/2026-06/2026-06-11t024225-testing-a-security-tool-like-it-can-hurt-people) makes the case for "red runs" — tests that verify the system fails loudly when it overclaims certainty. Meanwhile, ["Code Smells when you get AI to write your Frontend Tests"](/reading/2026-06/2026-06-22t185420-code-smells-when-you-get-ai-to-write-your-frontend-tests) catalogs specific failure modes AI tools introduce: over-mocking, happy-path bias, and writing tests that match a buggy implementation rather than intended behavior.

CI and infrastructure at scale gets concrete treatment in ["What CI Actually Looks Like at a 100-Person Team"](/reading/2026-04/2026-04-30t195531-what-ci-actually-looks-like-at-a-100-person-team), which describes an AI agent handling 575K weekly CI jobs at PostHog, tracing flaky tests to root causes and opening fix PRs. ["Approaching Zero Bugs"](/reading/2026-05/2026-05-02t094735-approaching-zero-bugs) is skeptical of the optimism: Daniel Stenberg's analysis of curl's vulnerability data finds no measurable sign yet that AI-assisted static analysis is bending the bug curve.

The AI-in-engineering debate is the most contested territory in these sources. Several argue AI tools are net negative when adopted without care. ["Agentic Coding is a Trap"](/reading/2026-04/2026-04-27t145041-agentic-coding-is-a-trap) warns that full agentic workflows cause skill atrophy, invert priorities toward speed over understanding, and create vendor lock-in. ["The Perils of AI to the Software Engineering Profession"](/reading/2026-05/2026-05-14t223612-the-perils-of-ai-to-the-software-engineering-profession) goes further, calling vibe coding categorically incompatible with safety-critical systems. ["The Lobster in the Hot Pot"](/reading/2026-05/2026-05-03t103944-the-lobster-in-the-hot-pot) frames the dependency risk as structural: gradual adoption erodes institutional knowledge before the costs become visible. ["Slow Mode"](/reading/2026-05/2026-05-19t193626-slow-mode) proposes a middle position, where an AI agent keeps the human programmer in the loop at every step to preserve learning.

Others focus on what has to change for AI tools to deliver their promised value. ["Why Most Developers Can't Use AI Effectively"](/reading/2026-05/2026-05-17t204925-why-most-developers-cant-use-ai-effectively) identifies structural barriers including weak type systems and organizational processes built for human-speed development. ["When Code is Cheap, Does Quality Still Matter"](/reading/2026-05/2026-05-22t091746-when-code-is-cheap-does-quality-still-matter) points out that AI lowers production cost but not ownership cost, and that LLMs can generate polished technical debt faster than any individual engineer. ["The Bottleneck Was Never the Code"](/reading/2026-05/2026-05-06t110728-the-bottleneck-was-never-the-code) reframes the whole question: the bottleneck was always organizational clarity, and agents amplify existing alignment or misalignment.

Knowledge and expertise have their own thread here. ["The Tacit Dimension"](/reading/2026-05/2026-05-19t110710-the-tacit-dimension-why-your-best-engineers-cant-tell-you) draws on Polanyi to argue that the most valuable engineering knowledge is structurally inaccessible to AI: pattern recognition and design intuition can only be transmitted through apprenticeship. ["Why Senior Developers Fail to Communicate Their Expertise"](/reading/2026-05/2026-05-13t060018-why-senior-developers-fail-to-communicate-their-expertise) argues the gap is not AI but language: engineers speak in complexity terms while the business thinks in uncertainty terms. ["The Unwritten Laws of Software Engineering"](/reading/2026-06/2026-06-10t073045-the-unwritten-laws-of-software-engineering) collects seven hard-won rules from production incidents, treating engineering judgment as a product of accumulated experience rather than derivable from first principles.

On hiring and onboarding, ["Learn Algorithms for Interviews, Forget Them for Work"](/reading/2026-04/2026-04-30t155134-learn-algorithms-for-interviews-forget-them-for-work) argues interview processes test a narrow trainable skill disconnected from production performance. ["The Systemic Decay of Tech Hiring"](/reading/2026-06/2026-06-22t182141-the-systemic-decay-of-tech-hiring) traces how error asymmetry and Goodhart's Law have made technical interviews progressively more dysfunctional. ["Your Onboarding Is a Hazing Ritual"](/reading/2026-05/2026-05-08t112608-your-onboarding-is-a-hazing-ritual-and-you-call-it-agile) extends the dysfunction to the first weeks on the job, where poor practices disguised as agile process systematically set new hires up to fail.

Version control gets two entries worth noting together: ["Reviewing Large Changes with Jujutsu"](/reading/2026-05/2026-05-31t164252-reviewing-large-changes-with-jujutsu) describes a concrete workflow for managing large PR reviews using jj's first-class conflict model, while ["The Git Commands I Run Before Reading Any Code"](/reading/2026-06/2026-06-18t024208-the-git-commands-i-run-before-reading-any-code) shows how log analysis alone can surface codebase risk before a single file is opened.

Security surfaces in ["Supply-chain Attack Using Invisible Unicode Code"](/reading/2026-04/2026-04-30t231634-supply-chain-attack-using-invisible-code-hits-github-and), which describes 151 malicious npm and GitHub packages encoding payloads in invisible Unicode variation-selector characters undetectable by static analysis. It is a reminder that engineering discipline includes the supply chain, not just the code you write.

The discipline is not static. Formal methods, long considered too expensive for most contexts, are being reconsidered: ["Formal Methods and the Future of Programming"](/reading/2026-06/2026-06-15t021106-formal-methods-and-the-future-of-programming) argues that agentic coding has lowered the cost of writing proofs enough to make verification newly cost-effective, and created demand for guarantees that tests alone cannot provide.
