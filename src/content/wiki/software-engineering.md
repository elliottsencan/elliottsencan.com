Software engineering is less a set of techniques than a discipline of tradeoffs: choosing what to build, how to structure it, how to verify it holds together, and how to maintain it over time. The sources tagged here span enough ground that no single thesis covers all of them, but a few tensions recur.

The oldest tension is between shipping and understanding. [Fagner Brack](/reading/2026-04/2026-04-30t155134-learn-algorithms-for-interviews-forget-them-for-work) argues that algorithm interview culture selects for a narrow, trainable skill while real engineering requires reading tradeoffs, shipping incrementally, and building systems that handle messy, unbounded real-world inputs. [Anton Zaides](/reading/2026-07/2026-07-07t170607-the-software-engineering-war) frames this as a live organizational conflict between "builders" who prioritize speed and "keepers" who insist on quality, noting that which camp you appear to belong to often depends on who is in the room.

Design principles sit at the center of most debates about code quality. [Henrique Teixeira](/reading/2026-06/2026-06-04t073318-single-responsibility-the-distorted-principle) shows that the Single Responsibility Principle is routinely misread as "do only one thing" when it actually means cohesive grouping under a single accountable responsibility; over-granularizing classes violates the cognitive simplicity SRP is meant to provide. [Kobi Hari](/reading/2026-04/2026-04-30t232001-a-better-way-to-build-angular-components-from-inputs-to) makes a parallel argument for component design: Angular components bloated with dozens of inputs should be refactored using the Composite Components pattern so concerns stay encapsulated. [Go Monk](/reading/2026-05/2026-05-04t231343-ai-likes-deep-modules) argues that deep modules — small interfaces hiding large implementations — reduce complexity for both human readers and LLMs. The common thread is that good abstraction compresses cognitive load; bad abstraction spreads it.

Testing appears across multiple sources as a place where discipline breaks down in predictable ways. [The Currents team](/reading/2026-05/2026-05-05t135218-designing-playwright-tests-that-survive-ui-refactors) argues that Playwright suites break during UI refactors not because of poor selector choices alone, but because tests couple to implementation details — CSS classes, DOM structure, position — rather than semantic roles and accessible names. A companion piece on [staging vs. production testing](/reading/2026-05/2026-05-15t120337-playwright-testing-in-staging-vs-production) frames the environment split as a deliberate decision with operational costs in both directions. AI-generated tests introduce their own failure modes: [How To Test Frontend](/reading/2026-06/2026-06-22t185420-code-smells-when-you-get-ai-to-write-your-frontend-tests) documents patterns like over-mocking, testing only happy paths, and writing tests to match a buggy implementation rather than intended behavior.

Security is an engineering concern, not an afterthought. [Dan Goodin's report](/reading/2026-04/2026-04-30t231634-supply-chain-attack-using-invisible-code-hits-github-and) on the invisible Unicode supply-chain attack — 151 malicious npm packages encoding payloads in variation-selector characters invisible to code review and static analysis — illustrates that attackers find attack surfaces in the assumptions developers make about what code looks like. [Emphere's testing writeup](/reading/2026-06/2026-06-11t024225-testing-a-security-tool-like-it-can-hurt-people) makes the related point that a security tool must be tested with red runs that prove it fails loudly when it overclaims certainty, not just green runs that confirm it works correctly.

Performance optimization is concrete and measured. [Arthur Pastel](/reading/2026-05/2026-05-14t151252-5-faster-fastblur-in-image-rs) demonstrates a 5.9x speedup on Rust's image-rs fast\_blur by replacing float arithmetic with integer accumulators and costly division with reciprocal multiplication. [Colin Breck](/reading/2026-06/2026-06-30t185207-when-impressive-performance-gains-do-not-matter) balances this with the observation that even order-of-magnitude improvements often fail to change outcomes because of attention thresholds, discrete capacity increments, and pipeline backpressure downstream.

Version control and codebase legibility are recurring practical concerns. [Ben Gesoff](/reading/2026-05/2026-05-31t164252-reviewing-large-changes-with-jujutsu) describes a Jujutsu workflow for reviewing large pull requests by squashing files into an empty parent commit as you read them, persisting review progress in version control. Ally Piechowski proposes five git log commands — churn hotspots, bus factor, bug clusters, velocity trends, firefighting frequency — that diagnose a new codebase's risks before opening a file.

Professional practice and tacit knowledge are harder to systematize. [Cekrem](/reading/2026-05/2026-05-19t110710-the-tacit-dimension-why-your-best-engineers-cant-tell-you) draws on Polanyi to argue that pattern recognition, unwritten conventions, and design intuition are structurally inaccessible to AI tools and transmissible only through apprenticeship. [Tuhin Nair](/reading/2026-05/2026-05-13t060018-why-senior-developers-fail-to-communicate-their-expertise) argues that the real challenge for senior developers is translating complexity management into the language of uncertainty reduction that the rest of the business speaks. [Anton Zaides's collection of hard-won rules](/reading/2026-06/2026-06-10t073045-the-unwritten-laws-of-software-engineering) — roll back before debugging, treat every external dependency as a future outage — reflects the same kind of embodied knowledge that resists formalization.

AI's effect on the discipline is contested. [Lars Faye](/reading/2026-04/2026-04-27t145041-agentic-coding-is-a-trap) argues that full agentic workflows accelerate skill atrophy and invert developer priorities toward speed over understanding. [Yusuf Aytas](/reading/2026-05/2026-05-22t091746-when-code-is-cheap-does-quality-still-matter) makes the complementary point that AI lowers the cost of producing code but not the cost of owning it. [Abednego Gomes](/reading/2026-05/2026-05-14t223612-the-perils-of-ai-to-the-software-engineering-profession) goes further, arguing that shipping AI-generated code without review is categorically incompatible with safety-critical systems. [Yaron Minsky at Jane Street](/reading/2026-06/2026-06-15t021106-formal-methods-and-the-future-of-programming) offers a different angle: agentic coding has made formal methods newly cost-effective, both by lowering the cost of writing proofs and by creating demand for verification that tests alone cannot satisfy. What the sources agree on is that judgment — about what to build, what to verify, what to own — remains non-delegable.
