Software engineering is not reducible to writing code. The sources collected here converge on a recurring theme: the technical substrate matters less than the judgment applied to it, whether that judgment belongs to an individual engineer, a team, or increasingly, an AI agent.

On design principles, ["AI Likes Deep Modules"](/reading/2026-05/2026-05-04t231343-ai-likes-deep-modules) argues that small interfaces hiding large implementations reduce cognitive load for humans and LLMs alike. ["Single Responsibility, the Distorted Principle"](/reading/2026-06/2026-06-04t073318-single-responsibility-the-distorted-principle) makes a related correction: SRP does not mean splitting code into the smallest possible units but grouping behaviors under a single accountable responsibility. Over-granularizing classes violates the very simplicity the principle was meant to provide. These two readings together point toward a consistent design heuristic: encapsulation is the goal, and both shallow modules and exploded class hierarchies undercut it.

The question of what engineers actually need to know professionally surfaces in ["Learn Algorithms for Interviews, Forget Them for Work"](/reading/2026-04/2026-04-30t155134-learn-algorithms-for-interviews-forget-them-for-work), which argues that algorithm interview tests measure a narrow, trainable skill with weak correlation to production performance. Real engineering means reading tradeoffs, shipping incrementally, and handling messy unbounded inputs. ["The Unwritten Laws of Software Engineering"](/reading/2026-06/2026-06-10t073045-the-unwritten-laws-of-software-engineering) distills complementary lessons from production incidents: roll back before debugging, treat every external dependency as a future outage. ["The Tacit Dimension"](/reading/2026-05/2026-05-19t110710-the-tacit-dimension-why-your-best-engineers-cant-tell-you) frames the deeper problem: the most valuable engineering expertise, pattern recognition and design intuition, is structurally inaccessible to documentation and can only be transmitted through apprenticeship.

Testing runs through many of these sources. ["Designing Playwright Tests That Survive UI Refactors"](/reading/2026-05/2026-05-05t135218-designing-playwright-tests-that-survive-ui-refactors) argues that fragile test suites result not from bad selectors alone but from coupling to implementation details rather than semantic roles. ["Playwright Testing in Staging vs Production"](/reading/2026-05/2026-05-15t120337-playwright-testing-in-staging-vs-production) provides a decision framework for where different test types belong. ["Code Smells when you get AI to write your Frontend Tests"](/reading/2026-06/2026-06-22t185420-code-smells-when-you-get-ai-to-write-your-frontend-tests) catalogs what goes wrong when AI generates those tests: over-mocking, happy-path bias, tests written to match a buggy implementation rather than intended behavior. ["Testing a Security Tool Like It Can Hurt People"](/reading/2026-06/2026-06-11t024225-testing-a-security-tool-like-it-can-hurt-people) takes the highest-stakes framing: a security tool that overclaims certainty is worse than one that abstains, so test infrastructure must prove the system fails loudly in adversarial conditions.

On the security side, ["Supply-chain attack using invisible Unicode code"](/reading/2026-04/2026-04-30t231634-supply-chain-attack-using-invisible-code-hits-github-and) describes 151 malicious npm and GitHub packages encoding payloads in invisible Unicode variation-selector characters, undetectable by code reviewers and static analysis tools. This directly implicates any workflow that accepts AI-generated or third-party code without careful review. ["YAML? That's Norway Problem"](/reading/2026-05/2026-05-18t113714-yaml-thats-norway-problem) illustrates a slower-moving failure mode: a spec bug where NO parses as false persists across popular libraries more than a decade after the fix landed in YAML 1.2.

The AI-in-the-workflow debate is the most contested terrain. ["Agentic Coding is a Trap"](/reading/2026-04/2026-04-27t145041-agentic-coding-is-a-trap) argues that fully autonomous coding agents accelerate skill atrophy and invert developer priorities toward speed over understanding. ["The Perils of AI to the Software Engineering Profession"](/reading/2026-05/2026-05-14t223612-the-perils-of-ai-to-the-software-engineering-profession) makes the sharpest version of this case: shipping AI-generated code without review is incompatible with safety-critical systems. ["Slow Mode"](/reading/2026-05/2026-05-19t193626-slow-mode) proposes a middle path: an AI coding agent that keeps the human involved at every step, trading short-term productivity for genuine learning and long-term ownership.

On the other side, ["When Code Is Cheap, Does Quality Still Matter?"](/reading/2026-05/2026-05-22t091746-when-code-is-cheap-does-quality-still-matter) concedes that AI lowers production cost but not ownership cost, and that LLMs can generate polished technical debt faster than any individual engineer. ["The bottleneck was never the code"](/reading/2026-05/2026-05-06t110728-the-bottleneck-was-never-the-code) reframes the question: coding agents make individual code-writing cheap, but the real bottleneck was always organizational, shared context and specification clarity, and agents amplify whatever alignment the organization already has. ["Formal Methods and the Future of Programming"](/reading/2026-06/2026-06-15t021106-formal-methods-and-the-future-of-programming) adds that agentic coding has made formal verification newly cost-effective, both by lowering the cost of writing proofs and by creating demand for tools that go beyond what tests alone can provide.

Version control and tooling round out the picture. ["Reviewing Large Changes with Jujutsu"](/reading/2026-05/2026-05-31t164252-reviewing-large-changes-with-jujutsu) describes a workflow using Jujutsu's first-class conflict handling to review large pull requests incrementally without losing state. ["The Git Commands I Run Before Reading Any Code"](/reading/2026-06/2026-06-18t024208-the-git-commands-i-run-before-reading-any-code) shows how churn hotspots, bus factor, and bug clusters extracted from git log can diagnose codebase risk before opening a single file. ["What CI Actually Looks Like at a 100-Person Team"](/reading/2026-04/2026-04-30t195531-what-ci-actually-looks-like-at-a-100-person-team) describes an AI agent handling CI triage across 575K weekly jobs, tracing flaky tests to root causes and opening fix PRs automatically.

Across all of these, the through-line is that engineering quality is not a property of any single tool or practice but of the accumulated judgment applied at every layer: design, testing, security, version control, and the human oversight that makes each of those layers trustworthy.
