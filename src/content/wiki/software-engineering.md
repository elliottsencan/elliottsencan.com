Software engineering is not a single skill but a cluster of practices connecting code quality, system design, tooling choices, and organizational behavior. The sources tagged here range widely, but a consistent argument runs through them: velocity and cleverness are less durable than judgment, legibility, and human accountability.

On code design, the debate over the Single Responsibility Principle illustrates a recurring failure mode. [Henrique Teixeira](/reading/2026-06/2026-06-04t073318-single-responsibility-the-distorted-principle) argues that SRP is routinely misread as "do one thing" when it actually means cohesive grouping under a single accountable concern, and that over-granularizing classes produces the very cognitive overhead the principle was meant to reduce. A related argument appears in the deep-modules framing: [Go Monk](/reading/2026-05/2026-05-04t231343-ai-likes-deep-modules) contends that small interfaces hiding large implementations reduce complexity for both humans and LLMs, contrasting these with shallow modules that leak implementation detail into every caller.

Code quality's relationship to cost is addressed directly by [Yusuf Aytas](/reading/2026-05/2026-05-22t091746-when-code-is-cheap-does-quality-still-matter): AI lowers the cost of producing code but not the cost of owning it. Taste and judgment remain necessary because LLMs can generate polished, well-formatted technical debt faster than any individual engineer. [Anton Zaides](/reading/2026-06/2026-06-10t073045-the-unwritten-laws-of-software-engineering) distills this into production-incident wisdom: roll back before debugging, treat every external dependency as a future outage. [Arthur Pastel](/reading/2026-05/2026-05-14t151252-5-faster-fastblur-in-image-rs) demonstrates the same principle at the algorithmic level, replacing float arithmetic with integer accumulators to achieve a 5.9x speedup in Rust's image-rs library.

Testing is a persistent stress point. [Fagner Brack](/reading/2026-04/2026-04-30t155134-learn-algorithms-for-interviews-forget-them-for-work) argues that algorithm interview performance weakly correlates with production competence; real engineering means reading tradeoffs and shipping incrementally against messy, unbounded inputs. On automated testing, [the Currents team](/reading/2026-05/2026-05-05t135218-designing-playwright-tests-that-survive-ui-refactors) shows that Playwright suites break during refactors not because of bad selectors alone but because tests couple to implementation details rather than semantic roles and accessible names. [Emphere Engineering](/reading/2026-06/2026-06-11t024225-testing-a-security-tool-like-it-can-hurt-people) takes testing rigor further, building a deterministic assurance platform with red runs that prove the system fails loudly rather than overclaims certainty. And [Daniel Stenberg](/reading/2026-05/2026-05-02t094735-approaching-zero-bugs) uses curl's vulnerability data to argue that despite powerful new static analysis tools, there is no measurable sign yet that open-source projects are approaching zero latent bugs.

Infrastructure and tooling form a significant surface area. [Sam Alba](/reading/2026-04/2026-04-30t195531-what-ci-actually-looks-like-at-a-100-person-team) describes Mendral's AI agent handling CI triage at PostHog's scale, ingesting billions of log lines and opening fix PRs automatically. [Ivan Velichko](/reading/2026-05/2026-05-04t231858-how-container-filesystem-works-building-a-docker-like) walks through assembling a Docker-like container from scratch using only Linux primitives to show how mount namespaces and root filesystem isolation actually work. [Ben Gesoff](/reading/2026-05/2026-05-31t164252-reviewing-large-changes-with-jujutsu) proposes a Jujutsu-based workflow for reviewing large pull requests incrementally, persisting progress in version control.

Security intersects with supply chain risk. [Dan Goodin](/reading/2026-04/2026-04-30t231634-supply-chain-attack-using-invisible-code-hits-github-and) reports that attackers encoded payloads in invisible Unicode variation-selector characters across 151 malicious npm packages, making them undetectable by code reviewers and static analysis tools.

Organizational and professional concerns recur throughout. [DHg](/reading/2026-05/2026-05-08t112608-your-onboarding-is-a-hazing-ritual-and-you-call-it-agile) documents how poor onboarding practices disguised as agile process systematically set new hires up to fail. [Tuhin Nair](/reading/2026-05/2026-05-13t060018-why-senior-developers-fail-to-communicate-their-expertise) argues that senior engineers speak in terms of complexity management while the rest of the business thinks in terms of uncertainty reduction, and that bridging that gap is the real challenge of expertise. [cekrem](/reading/2026-05/2026-05-19t110710-the-tacit-dimension-why-your-best-engineers-cant-tell-you) grounds this in Michael Polanyi's philosophy: the most valuable engineering knowledge, including pattern recognition and design intuition, is structurally inaccessible to AI and can only be transmitted through apprenticeship.

The sharpest tension across the sources is over AI coding tools. [Lars Faye](/reading/2026-04/2026-04-27t145041-agentic-coding-is-a-trap) argues that full agentic workflows accelerate skill atrophy and create vendor dependency, advocating instead for LLMs as secondary delegation tools. [Abednego Gomes](/reading/2026-05/2026-05-14t223612-the-perils-of-ai-to-the-software-engineering-profession) is harsher still, calling vibe coding categorically incompatible with safety-critical systems. [The Typical Set](/reading/2026-05/2026-05-06t110728-the-bottleneck-was-never-the-code) reframes the issue: coding agents make code-writing cheap, but the real bottleneck was always organizational alignment and specification clarity. [Yaron Minsky at Jane Street](/reading/2026-06/2026-06-15t021106-formal-methods-and-the-future-of-programming) offers a more optimistic angle, arguing that agentic coding has made formal verification newly cost-effective by lowering the cost of writing proofs. [Armin Ronacher](/reading/2026-06/2026-06-23t161552-the-coming-loop) is cautious: harness loops orchestrating agents are becoming unavoidable, but they amplify LLMs' worst tendencies, producing defensive and opaque code that may eventually require machine participation to maintain.
