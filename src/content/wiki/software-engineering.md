Software engineering resists any single definition because it spans concerns as different as algorithm selection, module design, version control workflows, CI infrastructure, and organizational communication. What the sources collected here share is a recurring theme: the gap between what looks like engineering competence and what actually produces good systems over time.

On design principles, the Single Responsibility Principle is a useful example of how foundational ideas get distorted in practice. [Teixeira argues](/reading/2026-06/2026-06-04t073318-single-responsibility-the-distorted-principle) that SRP is commonly misread as "do one thing" when it actually means grouping behaviors under a single accountable responsibility. Over-granularizing classes produces the same cognitive load SRP is supposed to eliminate. A parallel argument appears in module design: [Go Monk contends](/reading/2026-05/2026-05-04t231343-ai-likes-deep-modules) that deep modules — small interfaces hiding large implementations — reduce complexity for both humans and LLMs, in contrast to shallow modules that scatter concern across many tiny surfaces.

Testing is where design decisions become visible under pressure. [Currents Team shows](/reading/2026-05/2026-05-05t135218-designing-playwright-tests-that-survive-ui-refactors) that Playwright suites break during refactors not because of selector choices alone, but because tests couple to implementation details rather than semantic roles and accessible names. [AI-generated test code compounds this](/reading/2026-06/2026-06-22t185420-code-smells-when-you-get-ai-to-write-your-frontend-tests): over-mocking, happy-path-only coverage, and tests written to match a buggy implementation are documented patterns. CI scale introduces its own problems — [Sam Alba describes](/reading/2026-04/2026-04-30t195531-what-ci-actually-looks-like-at-a-100-person-team) PostHog running 575,000 weekly CI jobs and 33 million test executions, with an AI agent handling triage by ingesting billions of log lines and opening fix PRs automatically. Even at that scale, [Daniel Stenberg's curl data](/reading/2026-05/2026-05-02t094735-approaching-zero-bugs) suggests that AI-assisted static analysis has not produced a measurable reduction in latent bugs in open-source projects.

Code quality as a concept is under pressure from cheap generation. [Yusuf Aytas puts it plainly](/reading/2026-05/2026-05-22t091746-when-code-is-cheap-does-quality-still-matter): AI lowers the cost of producing code but not the cost of owning it, and LLMs can generate polished, well-formatted technical debt faster than any individual engineer. [Anton Zaides frames this as a cultural split](/reading/2026-07/2026-07-07t170607-the-software-engineering-war) between engineers who prioritize shipping fast and those who insist on quality, with position often determined by who is in the room rather than considered principle. [Abednego Gomes takes a harder stance](/reading/2026-05/2026-05-14t223612-the-perils-of-ai-to-the-software-engineering-profession), arguing that shipping AI-generated code without review is categorically incompatible with safety-critical systems.

Full agentic coding workflows attract the sharpest criticism. [Lars Faye argues](/reading/2026-04/2026-04-27t145041-agentic-coding-is-a-trap) they accelerate skill atrophy, invert developer priorities toward speed over understanding, and create vendor dependency. [Christoph Spörk extends this](/reading/2026-05/2026-05-03t103944-the-lobster-in-the-hot-pot) to institutional knowledge: dependency on LLMs erodes the accumulated expertise that organizations need to evaluate the output. [Val Town's Pete Millspaugh proposes](/reading/2026-05/2026-05-19t193626-slow-mode) a "Slow Mode" agent that keeps the programmer involved at every step, trading short-term productivity for genuine understanding. [Armin Ronacher acknowledges](/reading/2026-06/2026-06-23t161552-the-coming-loop) that harness loops are becoming unavoidable while warning they amplify LLMs' worst tendencies and risk creating codebases that require machine participation to maintain.

The expertise gap is not only technical. [Tuhin Nair identifies](/reading/2026-05/2026-05-13t060018-why-senior-developers-fail-to-communicate-their-expertise) that senior developers communicate in terms of complexity management while the business thinks in uncertainty reduction, and bridging that gap is as hard as any technical problem. [cekrem draws on Polanyi](/reading/2026-05/2026-05-19t110710-the-tacit-dimension-why-your-best-engineers-cant-tell-you) to argue that the most valuable engineering knowledge — pattern recognition, design intuition, unwritten conventions — is structurally inaccessible to AI tools and can only be transmitted through apprenticeship. [DHg points out](/reading/2026-05/2026-05-08t112608-your-onboarding-is-a-hazing-ritual-and-you-call-it-agile) that onboarding practices often destroy knowledge transfer before it begins, packing new hires with meetings and same-sprint workloads that make dysfunction invisible to management.

Practical tooling ranges from version control to shell hygiene. [Ben Gesoff describes](/reading/2026-05/2026-05-31t164252-reviewing-large-changes-with-jujutsu) a Jujutsu workflow for reviewing large pull requests by inserting an empty parent commit and squashing files into it as review proceeds, persisting progress in version control. [Ally Piechowski's git log commands](/reading/2026-06/2026-06-18t024208-the-git-commands-i-run-before-reading-any-code) — churn hotspots, bus factor, bug clusters — diagnose a codebase's risks before opening a single file. [Christian Hofstede-Kuhn catalogs](/reading/2026-04/2026-04-30t231815-shell-tricks-that-actually-make-life-easier-and-save-your) underused shell shortcuts and script safety flags that remain relevant across POSIX, Bash, and Zsh.

Security surfaces as a cross-cutting concern. [Dan Goodin reports](/reading/2026-04/2026-04-30t231634-supply-chain-attack-using-invisible-code-hits-github-and) on 151 malicious npm and GitHub packages encoding payloads in invisible Unicode variation-selector characters, undetectable by code reviewers and static analysis. The attack vector is the review process itself, which is the same process that AI code review is supposed to strengthen.

What runs through all of this is that the hardest parts of software engineering are not the parts that generate lines of code. They are the parts that determine whether those lines of code are correct, maintainable, secure, and understood by the people responsible for them.
