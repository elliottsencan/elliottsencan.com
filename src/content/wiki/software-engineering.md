Two recurring tensions define how software engineering is discussed here. The first is between demonstrated skill in controlled conditions and actual effectiveness in production. [Brack's essay](/reading/2026-04/2026-04-30t155134-learn-algorithms-for-interviews-forget-them-for-work) argues that algorithm interviews measure a trainable, narrow competency that has little to do with reading business context, handling messy inputs, or shipping incremental value. Passing a LeetCode screen and writing maintainable production code are related only loosely.

The second tension is between capability and comprehension. [Lars Faye](/reading/2026-04/2026-04-27t145041-agentic-coding-is-a-trap) makes the case that fully delegating coding to AI agents is self-defeating: the skills required to oversee and debug an agent's output are the same skills that atrophy when you stop writing code yourself. The result is compounding comprehension debt, along with practical risks like vendor lock-in and unpredictable token costs.

Both sources point toward the same underlying principle. Engineering judgment, the ability to evaluate outputs, question assumptions, and adapt to context, cannot be outsourced or skipped. Interviews that ignore it and tooling that bypasses it both create fragility downstream.
