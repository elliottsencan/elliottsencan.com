Software engineering is not a single skill but a cluster of practices held together by the question of how to build systems that work, stay working, and can be understood by people who didn't write them. The sources here converge on that question from many angles, and their collective tension is useful: AI tools promise to collapse the cost of producing code while several authors argue that production cost was never the real constraint.

The claim that "the bottleneck was never the code" comes through most directly in [The Typical Set's essay](/reading/2026-05/2026-05-06t110728-the-bottleneck-was-never-the-code), which argues that what coding agents actually amplify is organizational alignment — or its absence. Shared context, specification clarity, and management coherence determine output quality; agents just accelerate whatever dynamics already exist. Werner Vogels reaches a similar conclusion from a different angle: [AI-compressed prototyping](/reading/2026-06/2026-06-30t173037-a-return-to-two-pizza-culture) warrants amending Amazon's Working Backwards process, but the work of building things customers want remains the same.

What changes when code is cheap is the weight of judgment. [Yusuf Aytas](/reading/2026-05/2026-05-22t091746-when-code-is-cheap-does-quality-still-matter) observes that LLMs can generate polished, well-formatted technical debt faster than any individual engineer. [Abednego Gomes](/reading/2026-05/2026-05-14t223612-the-perils-of-ai-to-the-software-engineering-profession) argues that shipping AI-generated code without review is categorically incompatible with safety-critical systems. [Lars Faye](/reading/2026-04/2026-04-27t145041-agentic-coding-is-a-trap) adds that full agentic workflows invert developer priorities toward speed over understanding and accelerate skill atrophy. The counterproposal — from [Val Town's Pete Millspaugh](/reading/2026-05/2026-05-19t193626-slow-mode) — is a "Slow Mode" agent that keeps the human involved at every step, trading short-term throughput for genuine understanding and long-term ownership.

Skill atrophy matters because engineering expertise is not fully articulable. [Cekrem, drawing on Polanyi](/reading/2026-05/2026-05-19t110710-the-tacit-dimension-why-your-best-engineers-cant-tell-you), argues that the most valuable engineering knowledge — pattern recognition, design intuition, unwritten conventions — is structurally inaccessible to AI tools and can only be transmitted through apprenticeship. [Tuhin Nair](/reading/2026-05/2026-05-13t060018-why-senior-developers-fail-to-communicate-their-expertise) identifies a related gap: senior engineers communicate in terms of complexity management while the rest of the business thinks in terms of uncertainty reduction, and bridging that gap is itself a core engineering skill.

On the craft level, several sources push back against common distortions. [Henrique Teixeira](/reading/2026-06/2026-06-04t073318-single-responsibility-the-distorted-principle) argues that SRP is widely misread as "do one thing" when it actually means cohesive grouping under a single accountable responsibility — over-granularizing classes violates the cognitive simplicity SRP is meant to provide. [Go Monk](/reading/2026-05/2026-05-04t231343-ai-likes-deep-modules) argues for deep modules — small interfaces hiding large implementations — as a way to reduce complexity for both humans and LLMs. [Anton Zaides](/reading/2026-06/2026-06-10t073045-the-unwritten-laws-of-software-engineering) distills seven hard-won production rules, including rolling back before debugging and treating every external dependency as a future outage.

Testing and verification run through multiple sources. [Currents](/reading/2026-05/2026-05-05t135218-designing-playwright-tests-that-survive-ui-refactors) argues that Playwright suites break during refactors not because of bad selectors but because tests couple to implementation details rather than semantic roles. [How To Test Frontend](/reading/2026-06/2026-06-22t185420-code-smells-when-you-get-ai-to-write-your-frontend-tests) documents twenty-plus patterns AI tools introduce in generated tests — over-mocking, happy-path bias, testing the bug rather than the intent. [Imbue's experiment](/reading/2026-06/2026-06-23t212958-how-ai-code-review-can-make-correct-code-worse) found that AI review-and-fix pipelines cause weaker agents to break correct code by overreaching beyond review scope. [Daniel Stenberg](/reading/2026-05/2026-05-02t094735-approaching-zero-bugs), using curl's own data, finds no measurable sign yet that AI-assisted static analysis is driving open-source projects toward zero latent bugs. [Yaron Minsky at Jane Street](/reading/2026-06/2026-06-15t021106-formal-methods-and-the-future-of-programming) argues the reverse: agentic coding has made formal verification newly cost-effective precisely because tests alone can no longer provide sufficient confidence.

[Fagner Brack](/reading/2026-04/2026-04-30t155134-learn-algorithms-for-interviews-forget-them-for-work) identifies a structural mismatch in how engineers are hired: algorithm interviews test a narrow, trainable skill that weakly correlates with production performance, where real engineering means reading tradeoffs and building systems that handle messy, unbounded inputs. [Vladimir Klepov](/reading/2026-06/2026-06-22t182141-the-systemic-decay-of-tech-hiring) traces how error asymmetry, shared interviewer pools, and Goodhart's Law have made interviews progressively more dysfunctional. [DHg](/reading/2026-05/2026-05-08t112608-your-onboarding-is-a-hazing-ritual-and-you-call-it-agile) extends the critique to onboarding: packed calendars and same-sprint workloads from day one disguise dysfunction as agile process.

At the infrastructure layer, [Sam Alba](/reading/2026-04/2026-04-30t195531-what-ci-actually-looks-like-at-a-100-person-team) describes what CI triage looks like at scale — 575K weekly jobs, 33M test executions — when an AI agent ingests billions of log lines to trace flaky tests and open fix PRs. [Armin Ronacher](/reading/2026-06/2026-06-23t161552-the-coming-loop) warns that orchestrating coding agents in outer harness loops amplifies LLMs' worst tendencies and risks producing codebases that require machine participation to maintain. The security surface grows in parallel: [Dan Goodin](/reading/2026-04/2026-04-30t231634-supply-chain-attack-using-invisible-code-hits-github-and) reports attackers encoding payloads in invisible Unicode variation-selector characters across 151 npm and GitHub packages, undetectable by code review or static analysis.

The discipline's breadth — from shell ergonomics to formal verification, from CSS primitives to multi-agent harnesses — resists reduction to any single practice. What connects these sources is the insistence that producing code is the easy part, and that the hard part is always understanding what you built and why.
