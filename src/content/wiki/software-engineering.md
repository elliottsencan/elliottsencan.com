Software engineering resists clean definition because the field has always been a negotiation between technical rigor and human organization. The sources here cover that tension across several registers: what engineers actually need to know, how teams build and maintain systems, what tools the discipline relies on, and how AI is redrawing the boundary between human judgment and automated execution.

The gap between interview performance and on-the-job effectiveness points at a longstanding confusion about what software engineering is. [Fagner Brack](/reading/2026-04/2026-04-30t155134-learn-algorithms-for-interviews-forget-them-for-work) argues that algorithm interviews test a narrow, trainable skill with weak correlation to production performance, where the actual work is reading tradeoffs, shipping incrementally, and building systems that handle messy, unbounded real-world inputs. Hiring processes don't necessarily select for this. [Vladimir Klepov](/reading/2026-06/2026-06-22t182141-the-systemic-decay-of-tech-hiring) traces how error asymmetry drives ever-harder interview tests, shared interviewer pools destroy accountability, and candidates overfit the process via Goodhart's Law — an erosion produced by well-intentioned engineers.

Once someone is hired, the craft itself demands ongoing judgment that is difficult to articulate. [Cekrem](/reading/2026-05/2026-05-19t110710-the-tacit-dimension-why-your-best-engineers-cant-tell-you) draws on Polanyi to argue that the most valuable engineering expertise — pattern recognition, unwritten conventions, design intuition — is structurally inaccessible to AI tools and transmittable only through apprenticeship. [Tuhin Nair](/reading/2026-05/2026-05-13t060018-why-senior-developers-fail-to-communicate-their-expertise) identifies a parallel gap in communication: senior developers speak in complexity-management terms while the rest of the business thinks in terms of uncertainty reduction, and bridging that gap is itself a skill.

The technical practices around quality and reliability form a recurring thread. [Daniel Stenberg](/reading/2026-05/2026-05-02t094735-approaching-zero-bugs) uses curl's vulnerability data to argue that despite powerful new AI-assisted static analysis, there is no measurable sign that open-source projects are approaching zero latent bugs. [Anton Zaides](/reading/2026-06/2026-06-10t073045-the-unwritten-laws-of-software-engineering) distills hard-won rules from production incidents — roll back before debugging, treat every external dependency as a future outage — that no amount of tooling automates away. Testing practice is similarly contested: [the Currents team](/reading/2026-05/2026-05-05t135218-designing-playwright-tests-that-survive-ui-refactors) argues Playwright suites break during refactors because tests couple to implementation details rather than semantic roles, and [How To Test Frontend](/reading/2026-06/2026-06-22t185420-code-smells-when-you-get-ai-to-write-your-frontend-tests) documents 20-plus patterns AI tools introduce when generating tests, including over-mocking and writing tests to match a buggy implementation rather than intended behavior.

Module and component design runs through several sources. [Go Monk](/reading/2026-05/2026-05-04t231343-ai-likes-deep-modules) argues that deep modules — small interfaces hiding large implementations — reduce complexity for both humans and LLMs. [Kobi Hari](/reading/2026-04/2026-04-30t232001-a-better-way-to-build-angular-components-from-inputs-to) argues Angular components bloated with dozens of inputs should be refactored using composition patterns so each concern stays encapsulated. [Henrique Teixeira](/reading/2026-06/2026-06-04t073318-single-responsibility-the-distorted-principle) argues that the Single Responsibility Principle is widely misunderstood as "do only one thing" when it actually means cohesive grouping under a single accountable responsibility.

The AI question cuts across the whole discipline. [Lars Faye](/reading/2026-04/2026-04-27t145041-agentic-coding-is-a-trap) argues that full agentic coding workflows accelerate skill atrophy, invert developer priorities toward speed over understanding, and create vendor dependency. [Pete Millspaugh](/reading/2026-05/2026-05-19t193626-slow-mode) proposes a "Slow Mode" agent that keeps humans involved at every step, trading short-term productivity for genuine learning. [Yusuf Aytas](/reading/2026-05/2026-05-22t091746-when-code-is-cheap-does-quality-still-matter) observes that AI lowers the cost of producing code but not of owning it — LLMs can generate polished technical debt faster than any individual engineer. [Armin Ronacher](/reading/2026-06/2026-06-23t161552-the-coming-loop) warns that harness loops orchestrating coding agents amplify LLMs' worst tendencies and risk creating codebases that require machine participation to maintain. [Abednego Gomes](/reading/2026-05/2026-05-14t223612-the-perils-of-ai-to-the-software-engineering-profession) calls shipping unreviewed AI-generated code categorically incompatible with safety-critical systems.

Organizational factors shape outcomes as much as technical ones. [The Typical Set](/reading/2026-05/2026-05-06t110728-the-bottleneck-was-never-the-code) argues the real bottleneck was always shared context, specification clarity, and management coherence, and coding agents amplify whatever alignment or misalignment an organization already has. [DHg](/reading/2026-05/2026-05-08t112608-your-onboarding-is-a-hazing-ritual-and-you-call-it-agile) identifies poor onboarding practices disguised as agile process as a systemic failure that sets new hires up to fail while making dysfunction invisible to management.

Across the sources, a pattern holds: software engineering is a discipline in which technical choices and organizational choices are inseparable, and tools — including AI — change the cost of certain actions without removing the judgment required to make them wisely.
