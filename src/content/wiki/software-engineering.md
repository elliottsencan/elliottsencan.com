Software engineering is not a single discipline but a cluster of overlapping practices: writing and reviewing code, designing systems that hold together under load, testing behavior rather than implementation, managing dependencies, and shipping work inside organizations that have their own dysfunctions. The sources here span all of those layers, and several tensions recur across them.

The most persistent tension is between speed and understanding. [Lars Faye](/reading/2026-04/2026-04-27t145041-agentic-coding-is-a-trap) argues that fully agentic coding workflows accelerate skill atrophy and invert developer priorities toward output over comprehension, creating vendor dependency in the process. [Abednego Gomes](/reading/2026-05/2026-05-14t223612-the-perils-of-ai-to-the-software-engineering-profession) pushes further, calling "vibe coding" — shipping AI-generated code without review — categorically incompatible with safety-critical systems. [Yusuf Aytas](/reading/2026-05/2026-05-22t091746-when-code-is-cheap-does-quality-still-matter) frames the same problem economically: AI lowers the cost of producing code but not the cost of owning it, and LLMs can generate polished technical debt faster than any individual engineer. [Pete Millspaugh](/reading/2026-05/2026-05-19t193626-slow-mode) proposes a concrete alternative — a "Slow Mode" agent that involves the programmer at every planning step, trading short-term throughput for genuine ownership.

AI also changes what quality means downstream of writing code. [The Typical Set](/reading/2026-05/2026-05-06t110728-the-bottleneck-was-never-the-code) observes that coding agents make individual code-writing cheap but amplify whatever organizational misalignment already exists — the bottleneck was always shared context and specification clarity, not keystrokes. [Armin Ronacher](/reading/2026-06/2026-06-23t161552-the-coming-loop) warns that harness loops orchestrating agents tend to produce defensive, opaque code and risk creating codebases that require machine participation to maintain. [Imbue's experiment](/reading/2026-06/2026-06-23t212958-how-ai-code-review-can-make-correct-code-worse) found that AI reviewer-fixer pipelines can break correct code when weaker fixer models overreach beyond review scope. [Jaron Minsky at Jane Street](/reading/2026-06/2026-06-15t021106-formal-methods-and-the-future-of-programming) takes a different angle: agentic coding makes formal verification newly cost-effective precisely because it creates urgent demand for assurance that tests alone cannot provide.

Design principles remain foundational regardless of who writes the code. [Go Monk](/reading/2026-05/2026-05-04t231343-ai-likes-deep-modules) argues that deep modules — small interfaces hiding large implementations — reduce complexity for humans and LLMs alike. [Henrique Teixeira](/reading/2026-06/2026-06-04t073318-single-responsibility-the-distorted-principle) corrects the common misreading of the Single Responsibility Principle: it means cohesive grouping under one accountable responsibility, not fragmenting every function into its own class. [Kobi Hari](/reading/2026-04/2026-04-30t232001-a-better-way-to-build-angular-components-from-inputs-to) applies the same logic to Angular components, arguing that bloated input lists should be refactored using composition patterns so each concern stays encapsulated.

Testing surfaces its own set of recurring problems. [The Currents team on Playwright](/reading/2026-05/2026-05-05t135218-designing-playwright-tests-that-survive-ui-refactors) argues that test suites break during refactors not because of bad selectors but because tests couple to implementation details rather than semantic roles. [Their follow-on piece](/reading/2026-05/2026-05-15t120337-playwright-testing-in-staging-vs-production) frames staging-vs-production split as a decision about which flows can tolerate false negatives versus which require real-environment validation. [How To Test Frontend](/reading/2026-06/2026-06-22t185420-code-smells-when-you-get-ai-to-write-your-frontend-tests) catalogs over twenty patterns AI tools introduce in generated tests — over-mocking, happy-path-only coverage, and tests written to match a buggy implementation rather than intended behavior. [Daniel Stenberg](/reading/2026-05/2026-05-02t094735-approaching-zero-bugs) uses curl's own vulnerability data to argue that despite powerful AI-assisted static analysis, there is no measurable sign yet that open-source projects are approaching zero latent bugs.

Tooling and infrastructure choices accumulate into compounding advantages or liabilities. [Sam Alba](/reading/2026-04/2026-04-30t195531-what-ci-actually-looks-like-at-a-100-person-team) describes an AI triage agent handling 575,000 weekly CI jobs at PostHog's scale, tracing flaky tests to root causes and opening fix PRs automatically. [Anton Zaides](/reading/2026-06/2026-06-10t073045-the-unwritten-laws-of-software-engineering) distills seven production-hardened rules, including rolling back before debugging and treating every external dependency as a future outage. [Dan Goodin at Ars Technica](/reading/2026-04/2026-04-30t231634-supply-chain-attack-using-invisible-code-hits-github-and) documents a supply-chain attack encoding payloads in invisible Unicode variation-selector characters, undetectable by code reviewers and static analysis tools — a reminder that the dependency graph is a security surface.

Organizational dynamics determine whether good engineering practice survives contact with reality. [DHg](/reading/2026-05/2026-05-08t112608-your-onboarding-is-a-hazing-ritual-and-you-call-it-agile) argues that poor onboarding disguised as agile practice systematically sets new hires up to fail while making the dysfunction invisible to management. [Tuhin Nair](/reading/2026-05/2026-05-13t060018-why-senior-developers-fail-to-communicate-their-expertise) identifies a communication mismatch: senior engineers speak in complexity management terms while the rest of the business thinks in uncertainty reduction terms. [cekrem](/reading/2026-05/2026-05-19t110710-the-tacit-dimension-why-your-best-engineers-cant-tell-you) draws on Polanyi's philosophy of tacit knowledge to argue that the most valuable engineering expertise — pattern recognition, design intuition — is structurally inaccessible to AI tools and transmissible only through apprenticeship.

[Fagner Brack](/reading/2026-04/2026-04-30t155134-learn-algorithms-for-interviews-forget-them-for-work) closes the loop on hiring: algorithm interviews test a narrow, trainable skill that weakly correlates with production performance, where real engineering requires reading tradeoffs, shipping incrementally, and handling messy unbounded inputs. [Vladimir Klepov](/reading/2026-06/2026-06-22t182141-the-systemic-decay-of-tech-hiring) traces how well-intentioned engineers created dysfunctional interview processes through error asymmetry, shared interviewer pools, and Goodhart's Law — candidates overfit to the process, and the process stops measuring what it was meant to measure.
