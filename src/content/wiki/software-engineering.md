The practice of software engineering resists a single definition because it bundles together concerns that rarely resolve in the same direction: write correct code, ship it fast, keep it maintainable, secure it against attack, and coordinate the humans doing all of the above. The sources here collectively surface a recurring tension: AI tools amplify both the best and worst tendencies already present in teams and codebases.

On design, [a piece on deep modules](/reading/2026-05/2026-05-04t231343-ai-likes-deep-modules) argues that small interfaces hiding large implementations reduce complexity for both human and machine readers, and that LLMs find deep modules easier to reason about than shallow ones. [A refactoring guide for Angular components](/reading/2026-04/2026-04-30t232001-a-better-way-to-build-angular-components-from-inputs-to) makes a compatible argument in framework terms: components bloated with dozens of inputs should be decomposed into directives and sub-components so each concern stays encapsulated. [A clarification of the Single Responsibility Principle](/reading/2026-06/2026-06-04t073318-single-responsibility-the-distorted-principle) pushes back on the popular reading that SRP means "do one thing," arguing the principle is really about cohesive grouping under a single accountable responsibility — over-granularizing violates the cognitive simplicity SRP is meant to provide.

On tooling and workflow, [a survey of shell tricks](/reading/2026-04/2026-04-30t231815-shell-tricks-that-actually-make-life-easier-and-save-your) covers Readline key bindings, history search, and script safety flags that make daily work more reliable. [A practical SSH guide](/reading/2026-05/2026-05-04t231548-using-ssh-keys-to-make-connectivity-simpler-and-secure) covers key generation, agent forwarding, and commit signing. [A Jujutsu workflow for large PRs](/reading/2026-05/2026-05-31t164252-reviewing-large-changes-with-jujutsu) describes a technique of duplicating a change, inserting an empty parent commit, and squashing files into it as review progresses — persisting review state in version control without Git stashes. [A wishlist for code forges](/reading/2026-06/2026-06-23t231556-if-i-could-make-my-own-github) identifies gaps in current platforms: pre-commit remote CI, stacked PRs as first-class citizens, nuanced approvals, and a smaller self-hostable unit.

On testing and quality, [a Playwright guide](/reading/2026-05/2026-05-05t135218-designing-playwright-tests-that-survive-ui-refactors) finds that test suites break during UI refactors not primarily because of selector choices but because tests couple to implementation details rather than semantic roles and accessible names. [A staging vs. production framework](/reading/2026-05/2026-05-15t120337-playwright-testing-in-staging-vs-production) adds operational detail on which flows belong where. [A catalog of AI-generated test smells](/reading/2026-06/2026-06-22t185420-code-smells-when-you-get-ai-to-write-your-frontend-tests) documents over twenty recurring patterns — over-mocking, happy-path bias, tests written to match buggy implementations rather than intended behavior. [Research on AI code review pipelines](/reading/2026-06/2026-06-23t212958-how-ai-code-review-can-make-correct-code-worse) finds that weaker fixer agents in an implementer-reviewer-fixer chain overreach beyond review scope and break correct code, with softer fixer instructions eliminating catastrophic regressions.

On security, [a supply-chain attack report](/reading/2026-04/2026-04-30t231634-supply-chain-attack-using-invisible-code-hits-github-and) describes 151 malicious npm packages encoding payloads in invisible Unicode variation-selector characters, undetectable by code review or static analysis. [A security testing post](/reading/2026-06/2026-06-11t024225-testing-a-security-tool-like-it-can-hurt-people) argues that security tools need fixture invariants and red runs that prove the system fails loudly rather than overclaims certainty. [Daniel Stenberg's analysis of curl's vulnerability data](/reading/2026-05/2026-05-02t094735-approaching-zero-bugs) finds no measurable evidence that AI-assisted static analysis is moving open-source projects toward zero latent bugs.

On the AI coding question specifically, the sources disagree in emphasis but not in direction. [Lars Faye's critique of agentic coding](/reading/2026-04/2026-04-27t145041-agentic-coding-is-a-trap) argues full agentic workflows accelerate skill atrophy and invert developer priorities toward speed over understanding. [A Val Town proposal for Slow Mode](/reading/2026-05/2026-05-19t193626-slow-mode) suggests keeping the human involved at every step to trade short-term output for genuine learning. [An essay on code quality under AI](/reading/2026-05/2026-05-22t091746-when-code-is-cheap-does-quality-still-matter) notes that LLMs can generate polished technical debt faster than any individual engineer, making taste and judgment more important, not less. [Armin Ronacher's piece on agent harness loops](/reading/2026-06/2026-06-23t161552-the-coming-loop) warns that orchestration loops amplify LLMs' worst tendencies toward defensive, opaque code, risking codebases that require machine participation to maintain. Countering pure pessimism, [Jane Street's Yaron Minsky](/reading/2026-06/2026-06-15t021106-formal-methods-and-the-future-of-programming) argues agentic coding makes formal verification newly cost-effective, lowering proof-writing costs while raising demand for tools that go beyond what tests alone can provide.

Organizational concerns run through several pieces. [An essay on coding agent bottlenecks](/reading/2026-05/2026-05-06t110728-the-bottleneck-was-never-the-code) argues the real constraint was always shared context, specification clarity, and management coherence — agents amplify whatever alignment or misalignment already exists. [A piece on senior developer communication](/reading/2026-05/2026-05-13t060018-why-senior-developers-fail-to-communicate-their-expertise) finds that senior engineers speak in terms of complexity management while the rest of the business thinks in terms of uncertainty reduction, and bridging that gap is the real challenge of expertise. [An analysis of tacit knowledge](/reading/2026-05/2026-05-19t110710-the-tacit-dimension-why-your-best-engineers-cant-tell-you) draws on Michael Polanyi to argue that the most valuable engineering knowledge — pattern recognition, design intuition, unwritten conventions — is structurally inaccessible to AI and only transmissible through apprenticeship. [An onboarding critique](/reading/2026-05/2026-05-08t112608-your-onboarding-is-a-hazing-ritual-and-you-call-it-agile) shows how poor practices disguised as agile process systematically set new hires up to fail while making the dysfunction invisible to management.

Algorithm interview culture gets its own treatment: [Fagner Brack's piece](/reading/2026-04/2026-04-30t155134-learn-algorithms-for-interviews-forget-them-for-work) argues that algorithm interviews test a narrow, trainable skill that weakly correlates with production performance, where real engineering requires reading tradeoffs, shipping incrementally, and handling messy unbounded inputs. [A hiring manager's retrospective](/reading/2026-06/2026-06-22t182141-the-systemic-decay-of-tech-hiring) traces how error asymmetry drives ever-harder tests and candidates overfit to the process via Goodhart's Law.
