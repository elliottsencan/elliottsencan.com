Two recurring tensions in software engineering come down to the same underlying problem: optimizing for a proxy metric instead of the actual work.

[Fagner Brack](/reading/2026-04/2026-04-30t155134-learn-algorithms-for-interviews-forget-them-for-work) argues that algorithm interviews measure a trainable, narrow skill with weak correlation to production performance. Real engineering involves reading business context, handling messy inputs, and shipping incremental value, none of which maps cleanly onto sorting a clean array under a timer. The interview ritual persists not because it predicts engineering quality but because it is legible and defensible to hiring committees.

The same pattern appears in how engineers reach for AI agents. [Lars Faye](/reading/2026-04/2026-04-27t145041-agentic-coding-is-a-trap) identifies a self-defeating loop in fully delegating coding to agents: the skills required to oversee and debug agent output are exactly the skills that atrophy when you stop writing and reasoning through code yourself. Compounding that, agentic workflows introduce vendor lock-in, unpredictable token costs, and comprehension debt that accumulates silently.

Both sources point toward the same corrective. Judgment, the ability to read context and understand what is actually happening in a system, cannot be outsourced to a process optimized for something easier to measure. Interview prep and agentic automation are useful in bounded ways, but treating either as a substitute for maintained engineering thinking produces engineers who can neither evaluate their tools nor course-correct when those tools fail.
