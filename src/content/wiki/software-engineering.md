The craft sits at the intersection of technical skill and organizational behavior. [Fagner Brack](/reading/2026-04/2026-04-30t155134-learn-algorithms-for-interviews-forget-them-for-work) makes the case directly: algorithm interviews test a narrow, trainable skill that correlates weakly with production performance. Real engineering means reading tradeoffs, shipping incrementally, and building systems that handle messy, unbounded real-world inputs. That framing — engineering as judgment under constraint — runs through most of what makes day-to-day software work difficult.

Principles like the Single Responsibility Principle get distorted in practice. [Henrique Teixeira](/reading/2026-06/2026-06-04t073318-single-responsibility-the-distorted-principle) argues SRP is widely misread as "do only one thing" when it actually means cohesive grouping of behaviors under a single accountable responsibility. Over-granularizing classes violates the cognitive simplicity SRP is meant to provide. Similarly, [Go Monk](/reading/2026-05/2026-05-04t231343-ai-likes-deep-modules) argues that deep modules — small interfaces hiding large implementations — reduce complexity for both humans and LLMs, contrasting them with shallow modules that push complexity outward.

Much of engineering expertise is tacit. [cekrem](/reading/2026-05/2026-05-19t110710-the-tacit-dimension-why-your-best-engineers-cant-tell-you) draws on Michael Polanyi's philosophy to argue that pattern recognition, unwritten conventions, and design intuition are structurally inaccessible to AI tools and can only be transmitted through apprenticeship. [Tuhin Nair](/reading/2026-05/2026-05-13t060018-why-senior-developers-fail-to-communicate-their-expertise) frames a related gap: senior developers communicate in terms of complexity management while the rest of the business thinks in uncertainty reduction, and bridging that gap is the real challenge of software expertise.

AI coding tools have become the dominant source of tension in how the profession thinks about itself. Several sources agree on the same core risk: that generating code cheaply does not lower the cost of owning it. [Yusuf Aytas](/reading/2026-05/2026-05-22t091746-when-code-is-cheap-does-quality-still-matter) puts it plainly — LLMs can generate polished, well-formatted technical debt faster than any individual engineer. [Lars Faye](/reading/2026-04/2026-04-27t145041-agentic-coding-is-a-trap) argues that full agentic workflows accelerate skill atrophy, invert developer priorities toward speed over understanding, and create vendor dependency. [Abednego Gomes](/reading/2026-05/2026-05-14t223612-the-perils-of-ai-to-the-software-engineering-profession) goes further, calling vibe coding — shipping AI-generated code without review or testing — categorically incompatible with safety-critical systems. Pete Millspaugh at Val Town [proposes a "Slow Mode"](/reading/2026-05/2026-05-19t193626-slow-mode) agent that keeps the human involved at every step, trading short-term productivity for genuine learning and long-term code ownership.

The organizational bottleneck argument complicates the productivity framing. [The Typical Set](/reading/2026-05/2026-05-06t110728-the-bottleneck-was-never-the-code) argues that coding agents make individual code-writing cheap, but the real bottleneck was always shared context, specification clarity, and management coherence — and agents amplify whatever alignment or misalignment an organization already has. [DHg](/reading/2026-05/2026-05-08t112608-your-onboarding-is-a-hazing-ritual-and-you-call-it-agile) documents how poor onboarding practices disguised as agile process systematically set new hires up to fail, making the dysfunction invisible to management.

Tooling and infrastructure choices carry their own engineering weight. [Sam Alba](/reading/2026-04/2026-04-30t195531-what-ci-actually-looks-like-at-a-100-person-team) describes how Mendral's AI agent handles CI triage at PostHog's scale — 575K weekly jobs and 33M test executions — by tracing flaky tests to root causes and opening fix PRs automatically. The [Currents team on Playwright](/reading/2026-05/2026-05-05t135218-designing-playwright-tests-that-survive-ui-refactors) argues that test suites break during UI refactors not because of bad selector choices but because tests couple to implementation details rather than semantic roles and accessible names. Security is part of the infrastructure concern too: [Dan Goodin](/reading/2026-04/2026-04-30t231634-supply-chain-attack-using-invisible-code-hits-github-and) reports on 151 malicious npm packages encoding payloads in invisible Unicode variation-selector characters — undetectable by code reviewers and static analysis while remaining executable at runtime.

[Anton Zaides](/reading/2026-06/2026-06-10t073045-the-unwritten-laws-of-software-engineering) distills seven hard-won engineering rules from production incidents: roll back before debugging, treat every external dependency as a future outage. [Ally Piechowski](/reading/2026-06/2026-06-18t024208-the-git-commands-i-run-before-reading-any-code) offers a complementary set of git log commands — churn hotspots, bus factor, bug clusters — that diagnose a codebase's risks before opening a single file. [Daniel Stenberg](/reading/2026-05/2026-05-02t094735-approaching-zero-bugs) uses curl's vulnerability data to argue that despite powerful new AI-assisted static analysis, there is no measurable sign yet that open-source projects are approaching zero latent bugs.

Formal methods may be having a moment. [Yaron Minsky at Jane Street](/reading/2026-06/2026-06-15t021106-formal-methods-and-the-future-of-programming) argues that agentic coding has made formal verification newly cost-effective — lowering the cost of writing proofs while creating urgent demand for tools that go beyond what tests alone can provide. The concern about machine-generated code requiring machine participation to maintain is made explicit by [Armin Ronacher](/reading/2026-06/2026-06-23t161552-the-coming-loop), who warns that agentic harness loops amplify LLMs' worst tendencies — defensive, opaque code — and risk creating codebases that require ongoing machine involvement, raising urgent questions about human oversight and engineering judgment.
