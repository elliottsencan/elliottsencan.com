LLM engineering sits at the intersection of model-level work and systems-level work. At the model layer, tools like [Unsloth](/reading/2026-04/2026-04-24t093356-unsloth) compress the cost of fine-tuning substantially, offering up to 30x faster training and 90% less memory than FlashAttention 2 via custom kernels with LoRA and FP8 support. The [how-to-train-your-gpt](/reading/2026-05/2026-05-06t173338-raiyanyahyahow-to-train-your-gpt) interactive textbook grounds that work conceptually, walking through tokenization, rotary embeddings, attention, and inference from first principles. These tools lower the barrier to building domain-specific models, and the BARRED framework from [Plurai](/reading/2026-04/2026-04-28t140203-vibe-training-auto-train-a-small-language-model-for-your) demonstrates one payoff: synthetic training data generated by multi-agent debate can produce small classifiers that outperform GPT-4.1 on policy enforcement tasks at a fraction of the cost.

On the inference side, the engineering choices compound quickly. [CanItRun](/reading/2026-04/2026-04-29t173553-canitrun-can-my-gpu-run-this-llm) addresses the most basic constraint: VRAM budgets and quantization tradeoffs before a model ever runs. Once running, KV cache management becomes a primary cost lever. Two pieces from Everpure Engineering detail how treating the KV cache as a persistent, shared data asset rather than a per-request recomputation can cut prefill costs by up to 20x and reduce time-to-first-token in RAG workloads via granular prompt segmentation: [KV caching](/reading/2026-05/2026-05-20t073125-how-to-cut-llm-inference-costs-with-kv-caching) and [granular-prompt caching](/reading/2026-05/2026-05-20t073144-maximizing-llm-efficiency-granular-prompt-caching-with-pure). Inference routing adds another dimension: [DigitalOcean's Inference Router](/reading/2026-06/2026-06-21t192306-how-we-built-digitalocean-inference-router) uses a 30B MoE routing model to match requests to models on cost, latency, or quality, and [Arch-Router](/reading/2026-06/2026-06-21t192506-arch-router-aligning-llm-routing-with-human-preferences) proposes a compact 1.5B preference-aligned routing model that can accommodate new models without retraining. [What is Inference Engineering](/reading/2026-06/2026-06-21t130559-what-is-inference-engineering) synthesizes the techniques — quantization, speculative decoding, parallelism, disaggregation — as a distinct engineering discipline with its own career path.

At the application layer, agent architecture dominates current discussion. The [12-factor-agents](/reading/2026-05/2026-05-19t174452-humanlayer12-factor-agents) principle of unifying execution and business state into a single context-window-derived thread simplifies serialization, debugging, and recovery. [Harness design](/reading/2026-05/2026-05-18t221205-walkinglabslearn-harness-engineering) formalizes this further: the five subsystems (instructions, state, verification, scope, session lifecycle) are what convert model output into reliable engineering results. Anthropic's own GAN-inspired planner-generator-evaluator architecture, described in [Harness Design for Long-Running Application Development](/reading/2026-05/2026-05-01t104137-harness-design-for-long-running-application-development), demonstrates how multi-agent harnesses can sustain multi-hour coding sessions. [Agent observability](/reading/2026-05/2026-05-10t140531-agent-observability-needs-feedback-to-power-learning) extends this: traces alone do not improve systems; attaching feedback signals (ratings, indirect behavior, LLM-as-judge, deterministic rules) is what turns observability into a learning loop.

Retrieval is another contested layer. [PageIndex](/reading/2026-05/2026-05-06t171355-vectifyaipageindex) builds hierarchical tree indexes from long documents and uses LLM reasoning rather than vector similarity for retrieval, reaching 98.7% accuracy on FinanceBench. The Karpathy LLM Wiki pattern takes a different approach: an LLM compiles and maintains structured Markdown files that can be queried at scale without RAG at all. Two threads examine it honestly. The [practical guide](/reading/2026-04/2026-04-30t232052-how-to-implement-karpathys-llm-knowledge-base) covers ingest, maintenance, and health checks. The [honest takeaways post](/reading/2026-04/2026-04-30t232201-building-karpathys-llm-wiki-honest-takeaways) reports that cross-document synthesis is genuinely stronger than RAG for curated research, but hallucinations baked in at ingest propagate structurally, making the lint step non-negotiable.

Several sources push back on uncritical adoption. [Agentic Coding is a Trap](/reading/2026-04/2026-04-27t145041-agentic-coding-is-a-trap) argues that fully delegating implementation accelerates skill atrophy and creates vendor dependency. [When Code is Cheap](/reading/2026-05/2026-05-22t091746-when-code-is-cheap-does-quality-still-matter) notes that AI lowers production cost but not ownership cost, and that LLMs can generate technical debt faster than any individual engineer. Benchmark results temper enthusiasm at the capability boundary too: LLMs score near-perfect on TLA+ syntax but only ~46% on conformance to actual system behavior, [per SysMoBench](/reading/2026-05/2026-05-08t175639-can-llms-model-real-world-systems-in-tla), and the [Opus 4.7 reasoning curve study](/reading/2026-05/2026-05-14t190300-opus-47-low-vs-medium-vs-high-vs-xhigh-vs-max-the-reasoning) finds that maximum reasoning effort does not monotonically improve output quality, with medium effort winning on pass rate and cost across 29 real tasks. Sycophancy compounds these risks: [Bayesian modeling](/reading/2026-05/2026-05-03t103643-sycophantic-chatbots-cause-delusional-spiraling-even-in) shows that even ideally rational users spiral into delusional belief when models confirm rather than correct them, and neither eliminating hallucinations nor disclosing sycophancy fully prevents the effect.
